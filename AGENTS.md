# AGENTS.md

Guidance for AI coding agents (and humans) working in this repository.

## What this project is

`license-check` is a CLI + GitHub Action that scans an npm project's installed
dependency tree, identifies each dependency's license, categorises it, and flags
policy violations and missing licenses. It is **offline by default** and has
**zero runtime dependencies**.

It is an **informational** tool, **not legal advice**. Any user-facing output or
documentation you add must preserve that framing — do not imply the tool makes
compliance decisions or replaces a lawyer.

## Golden rules

- **No network access at runtime.** Detection reads the local filesystem only
  (`node_modules/*/package.json` and bundled `LICENSE` files). Do not add
  registry lookups or telemetry to the core scan path.
- **Keep runtime dependencies at zero.** New `dependencies` in `package.json`
  need a very good reason. `devDependencies` are fine.
- **Never execute dependency code.** We read files; we do not `require()` or run
  installed packages.
- **Don't fabricate adoption.** No fake stars/downloads/users in docs.

## Architecture (read these in order)

| File | Responsibility |
| --- | --- |
| `src/types.ts` | Shared types (`PackageRecord`, `PolicyConfig`, `LicenseCategory`, …). |
| `src/spdx.ts` | Bundled SPDX id → category map; expression parser/evaluator. |
| `src/detect.ts` | Detect a license for one package dir (manifest fields → LICENSE file). |
| `src/walk.ts` | Resolve & walk the dependency tree from a root `package.json`. |
| `src/policy.ts` | Load/validate a policy file; evaluate packages against it. |
| `src/format.ts` | Render table / JSON / Markdown output. No formatting deps. |
| `src/cli.ts` | Arg parsing + command dispatch. Core logic is in `run(argv, io)`. |
| `src/index.ts` | Public library API (re-exports). |

`run(argv, io)` in `src/cli.ts` is pure with respect to injected I/O (`cwd` +
write callbacks) and returns an exit code. **Add CLI behaviour there**, not in
the `isMainModule()` bootstrap block, so it stays unit-testable.

## Commands

```bash
npm install           # install dev deps
npm test              # vitest run (the source of truth)
npm run test:watch    # watch mode
npm run typecheck     # tsc --noEmit (strict)
npm run lint          # eslint
npm run build         # tsup -> dist/ (esm + d.ts, shebang preserved)
node dist/cli.js scan --dir test/fixtures/sample-project --no-color   # manual run
```

Before opening a PR, all of `npm run lint && npm run typecheck && npm test`
must pass. CI runs them on Node 20 and 22.

## Tests & fixtures

- Tests live in `test/*.test.ts` (Vitest). Every detection and policy path has
  coverage — keep it that way.
- Fixtures are **real files** under `test/fixtures/`, including fake
  `node_modules` trees. They are intentionally committed (note the anchored
  `/node_modules/` pattern in `.gitignore`).
- The fixtures are regenerated by `/tmp`-style generator scripts in PR history,
  but the committed files are the source of truth — edit them directly or add
  new ones. Cover new behaviour with a new fixture rather than mutating an
  existing one other tests depend on.
- `test/fixtures/sample-project` exercises: permissive, dual (`OR`), weak/strong/
  network copyleft, proprietary, unknown, deprecated object/array license forms,
  scoped packages, nested resolution, LICENSE-file detection, and missing
  licenses. `auto-project` exercises `.licensecheckrc.json` auto-detection.

## Conventions

- TypeScript, ESM, strict mode, `noUncheckedIndexedAccess` on. Respect it.
- Node built-ins use the `node:` prefix (`node:fs`, `node:path`).
- Keep `VERSION` in `src/cli.ts` in sync with `package.json` — a test enforces
  this.
- Prefer small, well-documented pure functions. Match the existing comment
  density (each module has a top-of-file doc comment explaining intent).
- SPDX ids use canonical short identifiers; add normalisation aliases in
  `NORMALIZATION_MAP` rather than scattering special cases.

## When adding a license

Add the SPDX id to `SPDX_CATEGORIES` in `src/spdx.ts` under the right group,
and, if it has a recognisable license-file header, add a signature to
`SIGNATURES` in `src/detect.ts`. Add/extend a test.

## Releasing

Bump `version` in `package.json` (and `VERSION` in `src/cli.ts`), update
`CHANGELOG.md`, tag `vX.Y.Z`. See `CONTRIBUTING.md`.
